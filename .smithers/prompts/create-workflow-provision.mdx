# Provision docs & skills for the new workflow

You are the **docs & skills** step of the "build me a Smithers workflow"
pipeline. Before anyone designs or writes the workflow, your job is to make sure
the right knowledge and capabilities are actually available, both for the
authoring agents (so the design is grounded in the real API, not guessed) and
for the worker agents the finished workflow will dispatch.

You have shell + file tools. **Actually run the commands and install what's
missing.** Do not just recommend. Then report what you did. Do not fabricate
results: if a command fails, record the failure in `notes` instead of inventing
its output.

## Do this

### 1. Pull the relevant Smithers docs into context

The docs are progressively disclosed as `llms-*.txt` fragments. Load only what
this workflow needs, not the whole bundle.

- `bunx smithers-orchestrator docs` prints the concise `llms.txt` index.
- Then pull the fragments that match this workflow's shape:
  `llms-core.txt` (runtime, JSX surface, CLI, components, recipes) is almost
  always relevant; add `llms-memory.txt`, `llms-openapi.txt`,
  `llms-observability.txt` (HTTP/gateway/MCP/OTel), `llms-integrations.txt`
  (agent runtimes, tools), or `llms-events.txt` only if the spec calls for them.
- Because every new workflow ships a custom UI, also skim the gateway-react
  material (docs on `smithers-orchestrator/gateway-react` and `smithers ui`) and
  1-2 existing UIs in `.smithers/ui/` (e.g. `.smithers/ui/vcs.tsx`,
  `.smithers/ui/create-workflow.tsx`) so the design step knows the hook surface.
- `bunx smithers-orchestrator ask "<focused question>"` for anything specific
  (e.g. "how do I add a human approval gate?", "how does Ralph decide to stop?").

Record each fragment you pulled and why in `docsFragments`.

### 2. Find the closest example to copy

Browse `examples/` (and `{props.workflowsDir}`) for the workflow(s) nearest to
this spec; they are the fastest correct starting point. `ls examples`, then read
1-3 candidates. Note them in `examples` with the reason each is a good template.

### 3. Pick the components to compose; do not hand-roll loops

Patterns seen twice in the field already ship as components. Identify which the
new workflow should reuse instead of re-implementing: `<ReviewLoop>`,
`<Optimizer>`, `<ScanFixVerify>`, `<Panel>`, `<Debate>`, `<Supervisor>`,
`<Saga>`, `<Kanban>`, `<MergeQueue>`, `<EscalationChain>`,
`<ClassifyAndRoute>`, `<GatherAndSynthesize>`, plus the local
`.smithers/components/*` (e.g. `ValidationLoop`, `Review`, `ForEachFeature`).
Record them in `components`.

### 4. Install the skills the worker agents will need

The finished workflow dispatches real agents (Claude Code, Codex, ...) to do the
work. Those agents are far more capable when the right **skills** are synced into
their agent dirs.

- Inventory what already exists: list `{props.skillsDir}` and note relevant
  existing skills as `present`.
- Decide which skills this workflow's tasks need given its domain (e.g. a deploy
  workflow needs infra/k8s skills; a docs workflow needs a docs skill; almost
  any authoring/orchestration workflow benefits from the `smithers` skill itself).
- Install the missing ones with `bunx smithers-orchestrator skills add <name>`
  (this syncs them into the agent skill dirs). Mark those `installed`.
- If a needed skill genuinely does not exist yet, mark it `recommended` and say
  what it should cover; the design step can note it as a follow-up.

Record every skill in `skills` with its `action` (installed | present |
recommended) and the reason.

### 5. Choose the agent pools

From `.smithers/agents.ts`, pick the narrowest role pool for every AI task. The
generated pools are ordered fallback chains: Codex runs first, and Claude/Kimi
are attempted only when every Codex candidate in that pool is unavailable or
fails. Do not put providers in parallel just to obtain fallback behavior.

- `agents.planning` uses GPT-5.6 Sol for graph architecture, orchestration,
  review, and other judgment-heavy work.
- `agents.research` uses GPT-5.6 Luna for source gathering and synthesis.
- `agents.implement` uses GPT-5.6 Luna for code writing, fixes, and other
  implementation tasks, even when they use file or shell tools.
- `agents.midTier` uses GPT-5.6 Terra for validation, tool-heavy checks, and
  ordinary mid-tier reasoning that is neither implementation nor top-tier
  planning/review.
- `agents.cheapFast` uses GPT-5.6 Luna for low-stakes classification,
  extraction, summarization, and other cheap high-volume work.

List the selected role pools in `agents`, along with which task will use each
one. The design manifest should use these five public roles (`planning`,
`research`, `implement`, `midTier`, `cheapFast`) rather than generic legacy
pool names.

## Acceptance criteria

- `docsFragments` names at least the fragments you actually read.
- `examples` names at least one real template file that exists on disk.
- Every `installed` skill was actually installed by a command you ran.

## Output

Return the provisioning manifest. `notes` should call out anything the design
step must know (a missing skill, an API gotcha you found in the docs, a template
to start from). Do not design or write the workflow here.
